# P0 健身新手首次训练体验设计

日期：2026-07-15  
状态：用户已确认方向，等待书面设计复核  
范围：仅处理《第一次使用体验审查》的 P0 问题；P1、P2 不在本轮实施范围

## 1. 背景与问题

当前产品具备训练模板、身体状态建议、训练草稿、历史记录、离线使用和数据导出等完整能力，但首次使用路径没有围绕一个明确任务组织。新用户同时看到今日建议、安全提示、状态卡、周目标、支持约定、新手引导、完整日常表单、技术状态和多个训练入口，难以快速判断应该从哪里开始。

当前训练保存还存在实际逻辑缺陷：系统通过“填写重量、修改模板次数或修改默认备注”推断一组是否完成。用户照着模板完成默认次数，或完成平板支撑、快走、自重训练后不修改预填值，可能仍被判断为没有完成任何一组，无法保存训练。

P0 改造必须把这些问题视为一条完整的新手旅程，而不是分别修改几张卡片：

> 看懂产品 → 开始训练 → 逐组完成 → 顺利保存

## 2. 产品目标

产品定位收束为：

> 给健身新手的每日训练决策助手：告诉用户今天练什么，并带用户逐组完成。

本轮将产品显示名称改为“今天练什么”。这是面向用户的描述性名称，不在 P0 内追加品牌命名项目。副标题统一为“根据恢复状态安排适合今天的训练，并带你逐组完成”。

本轮成功标准：

1. 陌生用户能在 5 秒内理解产品解决“今天练什么”的问题。
2. 用户能在应用打开后 3 秒内指出从哪里开始训练，且首屏只有一个视觉上最强的主按钮。
3. 从应用首页到第一组最多两次点击。
4. 新用户不需要理解完整表单、模板系统或 RPE 才能开始训练。
5. 模板默认值不修改、徒手动作不填写重量、按秒或按分钟动作均可明确完成并保存。
6. 训练总结字段只在训练结束后出现。
7. 账号、建议模式、离线缓存和安装状态不再占据正常首页。
8. 现有历史数据、模板、草稿、导入导出和离线能力保持可用。

## 3. 非目标

本轮不处理以下 P1、P2 内容：

- 五栏导航缩减；
- 第二天留存和完整训练完成页；
- 下次训练建议和渐进加重；
- 休息计时器、震动反馈和屏幕常亮；
- 分享、提醒和推送；
- 云同步、支付和 Pro 权益；
- 报告、洞察和长期复盘重构；
- 支持伙伴和周目标功能本身的增强。

周目标、支持约定、完整日常记录、数据报告、导入导出、账号说明、动作库和模板管理不会被删除，只会退出首次核心路径。

## 4. 核心设计决策

### 4.1 训练优先，身体状态可选调整

首页以“开始训练”为唯一主任务。用户即使尚未填写睡眠、精力和酸痛，也能使用明确标注为“新手默认方案”的基础训练。

四项身体状态通过次入口“用 4 个状态调整今天计划”收集。填写后，系统才将建议标注为“已根据今天状态调整”。身体状态是训练决策的输入，不再作为与训练并列的日记产品。

### 4.2 疼痛是唯一必要的首次安全门槛

如果用户当天没有任何疼痛状态，首次点击“开始训练”后只询问一个二选一问题：是否存在刺痛、关节痛或影响正常活动的异常疼痛。

- 选择“没有”：进入第一组。
- 选择“有”：进入四项状态面板，并将建议调整为恢复或保守方案。
- 当天已有疼痛记录：不重复询问，直接按已有状态进入相应方案。

这样既避免用完整问卷阻塞开练，也满足从首页到第一组最多两次点击的要求。

### 4.3 计划目标、实际结果和完成状态分离

模板中的重量、次数、秒数和提示属于计划目标。用户实际完成的数据属于实际结果。是否完成由显式状态决定，不能再通过“数据是否被修改”推断。

每组状态为：

- `pending`：待完成；
- `completed`：用户明确点击完成；
- `skipped`：用户明确跳过。

只有 `completed` 组进入正式训练记录。`pending` 和 `skipped` 只保留在训练草稿中。

### 4.4 当前组聚焦，同时保留完整计划

训练页面默认只突出当前动作和当前组，降低手机操作密度。完整计划可展开，允许用户因器械占用或临时安排自由切换动作、返回修改或跳过某组。

这不是严格锁步向导。系统提供默认下一步，但不阻止用户调整顺序。

### 4.5 正常状态保持安静，异常状态主动出现

账号、建议来源、离线缓存、安装和导出进入设置。正常首页不展示技术状态。

只有以下异常主动出现：

- 当前离线，但记录仍可保存在本机；
- 本地保存失败；
- 存储空间或数据损坏导致无法保存；
- 新版本已可用；
- 明显疼痛需要改变训练建议。

## 5. 组件职责边界

P0 实施应按用户旅程拆分职责，避免继续把首页、状态记录、训练执行和保存规则耦合在同一渲染流程中：

1. **Landing**：只渲染产品承诺、三步价值和进入应用的主按钮；不读取训练数据。
2. **Home Decision**：读取当天四项状态和已有训练，输出默认方案或状态调整后的方案；不负责编辑训练组。
3. **Quick Readiness**：只收集睡眠、精力、酸痛和疼痛，保存后请求 Home Decision 重新计算。
4. **Pain Gate**：仅在当天缺少疼痛状态时处理一次二选一安全确认；不扩展成完整问卷。
5. **Workout Session**：管理计划、当前组、实际结果、组状态、开始时间和草稿；不负责生成正式总结卡片。
6. **Workout Summary**：读取已完成组和会话时间，收集整体感受并提交正式训练记录。
7. **Settings Status**：承载账号、建议来源、离线、安装和导出状态；只将真正异常上报到全局提示。

这些单元通过明确的数据对象通信。Home Decision 输出训练计划；Workout Session 接收计划并输出已完成组；Workout Summary 将已完成组转换为现有正式训练记录。各单元的内部展示可以调整，但不能绕过这些边界重新通过 DOM 输入变化推断完成状态。

## 6. 页面与信息架构

### 6.1 根路径：极简落地页

根路径 `/` 只回答三个问题：产品做什么、对健身新手有什么帮助、从哪里开始。

推荐首屏内容：

> 今天练什么，直接告诉你  
> 根据睡眠、精力和身体状态，为健身新手安排适合今天的训练，并带你逐组完成。

主按钮为“看看今天怎么练”，进入 `/app/`。

落地页可以用一行表达核心循环：“状态判断 → 跟着训练 → 自动记录”，并保留一句可信说明：“数据默认保存在你的设备，可随时导出。”

落地页不展示功能清单、报告截图、Pro 权益、AI 额度、账号状态、路线图、周目标或支持伙伴。

### 6.2 应用路径：`/app/`

应用负责实际记录和训练。安装后的 PWA 直接打开 `/app/`，不回到营销落地页。

新用户首屏只保留：

1. 产品名称“今天练什么”；
2. 设置入口；
3. 一套今日训练方案；
4. 唯一主按钮“开始训练”；
5. 次入口“用 4 个状态调整今天计划”；
6. 有历史数据时才出现的小入口“使用上次训练”。

首次首屏移除或隐藏：

- 账号、AI、离线和安装状态标签；
- 安装方式和导出按钮；
- 周目标；
- 支持约定；
- 今日四项状态卡；
- 状态评分；
- 重复的新手引导卡；
- 完整日常记录表单；
- 空数据卡片；
- 默认展开的医疗免责声明；
- 相互竞争的多个训练和记录按钮。

## 7. 首页决策区

### 7.1 尚未填写身体状态

首页明确显示：

- “新手默认方案”；
- 训练名称，例如“全身入门”；
- 预计时长，例如“约 35 分钟”；
- 普通语言强度，例如“动作保持稳定，不需要做到力竭”；
- 主按钮“开始训练”；
- 次入口“用 4 个状态调整今天计划”；
- 安全提示链接“如有异常疼痛，请先调整计划”。

不得使用“已根据你的状态”之类暗示个性化的文案。

### 7.2 已填写身体状态

首页显示：

- “已根据今天状态调整”；
- 正常练、轻量练或恢复日；
- 调整后的模板、时长和普通语言强度；
- 主按钮“开始训练”或在明显疼痛时改为“查看恢复方案”。

“为什么这样建议”默认折叠，不在首屏展示长理由列表。

## 8. 四项快速状态面板

四项状态在一个手机屏幕内完成，使用大按钮而非要求用户理解精确量表：

1. 睡眠：少于 6 小时、6–7 小时、7–8 小时、8 小时以上；
2. 精力：很低、一般、不错、很充足；
3. 酸痛：没有、轻微、明显、很严重；
4. 异常疼痛：没有、有一点、明显疼痛。

用户点击“更新今天的训练”后，选择映射到现有数值模型并保存当天状态。饮水、心情、习惯和备注进入“补充今日状态”，不影响建议生成或开始训练。

明显疼痛必须优先生成恢复方案，并提示停止可能诱发疼痛的动作；持续或加重时建议咨询专业人士。

## 9. 训练会话

### 9.1 会话状态

训练会话采用明确状态流：

> 未开始 → 疼痛确认（必要时）→ 训练中 → 结束确认 → 训练总结 → 已保存

页面刷新或关闭时，未完成会话继续保持为“训练中”草稿，不自动生成正式训练记录。

### 9.2 进入训练

点击首页主按钮后，系统载入今日方案并直接显示第一组。开始前不展示日期、训练标题、预计或实际总时长、整体 RPE、备注、模板工具栏和完整动作表格。

日期自动使用当天，标题自动使用模板名，开始时间自动记录。

### 9.3 当前组界面

移动端首屏包括：

- 训练名称；
- 从 `startedAt` 计算的实际计时；
- 已完成组数与计划组数；
- 当前动作和当前组序号；
- 一句动作提示；
- 计划目标；
- 与动作类型匹配的实际结果控件；
- 主按钮“完成这一组”；
- 次入口“暂时跳过”和“更多”；
- 可展开的完整训练计划。

单组 RPE 和备注默认放入“更多”。普通用户不需要在每组理解或填写 RPE。

### 9.4 动作计量类型

模板和训练结果需要明确计量类型：

- `reps`：重复次数；
- `seconds`：秒；
- `minutes`：分钟；
- `completion`：仅记录完成。

重量始终可选。界面按类型显示“12 次”“30 秒”“10 分钟”或“完成即可”，不再用备注解释单位。

### 9.5 完成、撤销与跳过

点击“完成这一组”后：

1. 当前组明确标记为“已完成”；
2. 进度立即更新；
3. 自动进入下一待完成组；
4. 短时间显示“撤销”；
5. 完整计划中可随时返回修改。

修改已完成组的数据不会自动取消完成状态。只有用户点击“重新设为未完成”才撤销。

点击“暂时跳过”将状态设为 `skipped` 并进入下一组。跳过组不进入正式训练记录。

## 10. 结束与保存

### 10.1 结束确认

用户点击“结束训练”时：

- 若仍有未完成组，显示剩余数量，并提供“继续训练”和“提前结束”；
- 若没有任何已完成组，只能继续、保留草稿或放弃，不能保存空训练；
- 若至少完成一组，可进入训练总结。

### 10.2 训练总结

训练总结只显示：

- 自动计算的实际时长，可手动修正；
- 必选的整体感受：轻松、正合适、有点吃力；
- 可选展开的精确 RPE；
- 可选备注；
- 主按钮“保存训练”。

三档整体感受映射到现有整体 RPE 数值，但不得默认预选，以免伪造用户感受。

### 10.3 正式保存规则

正式训练记录只包含 `completed` 组。未修改的模板目标在用户点击完成后作为实际结果保存。徒手动作、按秒动作、按分钟动作和仅完成动作不需要填写重量。

保存成功后清除训练草稿并返回应用首页。P0 只需要清晰确认“训练已保存”，不增加 P1 的进步反馈、分享或下次提醒。

## 11. 数据与兼容性

### 11.1 草稿数据

训练草稿至少保存：

- 训练模板或标题；
- `startedAt`；
- 当前动作与当前组；
- 计划目标；
- 实际结果；
- 每组 `pending`、`completed` 或 `skipped` 状态；
- 可选单组 RPE 和备注。

草稿在每次输入、完成、撤销、跳过和切换组后持久化。

### 11.2 历史记录兼容

现有正式训练记录中的组均视为已完成，不重写历史数据。编辑旧训练时，载入的组初始状态为 `completed`。

旧版草稿保留原输入。旧结构没有持久化模板的默认次数和默认备注，因此迁移时不能可靠判断次数或备注是否被用户修改。只有存在重量的组标记为 `completed`，其余组统一标记为 `pending`；原次数、RPE 和备注仍完整显示，用户可以显式确认或修正。迁移后不再使用隐式推断。

导入导出需要保留新的计量类型。缺少类型的旧数据默认按 `reps` 读取，并继续兼容现有备注。

## 12. 设置与技术状态

页头只保留品牌和设置入口。现有账号、建议模式、离线、安装、导出能力移入设置，并用用户语言呈现：

- 数据保存在这台设备；
- 可以离线记录；
- 安装到主屏幕；
- 导出备份；
- 账号与同步；
- 建议来源。

P0 不缩减现有一级导航，因为导航缩减属于 P2。设置入口可以进入现有帮助或设置区域，不要求同时重构所有导航。

## 13. 静态路由与 PWA

静态发布结构调整为：

- `/`：落地页；
- `/app/`：应用；
- 隐私、条款和共享静态资源继续保留。

本地服务需要将 `/app` 和 `/app/` 映射到应用入口文件。GitHub Pages 使用真实的 `app/index.html` 支持目录访问。

Manifest 的 `start_url` 指向 `/app/` 对应的项目相对路径，安装后直接打开应用。Service Worker 同时缓存落地页、应用页和共享资源，并按导航目标选择离线回退：根路径回退落地页，应用路径回退应用页。不能用单一入口处理所有离线导航。

所有路径必须使用项目相对地址，兼容 GitHub Pages 的仓库子路径部署，不能写死站点根路径。

## 14. 错误与恢复

| 情况 | 行为 |
| --- | --- |
| 用户未完成任何组便结束 | 只能继续、保留草稿或放弃 |
| 还有未完成组便结束 | 显示剩余数量，允许继续或提前结束 |
| 本地保存失败 | 保留页面数据和草稿，显示重试入口 |
| 页面刷新或误关 | 恢复当前组、输入、完成状态和开始时间 |
| 推荐模板缺失 | 回退到基础新手方案，不显示空白页 |
| 明显疼痛 | 不载入普通训练，显示恢复方案 |
| 当前离线 | 允许训练和本地保存，只显示简短异常说明 |
| 旧历史记录 | 继续视为已完成，不改变历史内容 |

## 15. 可访问性与手机操作

- 当前组主按钮在常见手机首屏内可见；
- 所有可点击控件具有至少适合触摸的尺寸；
- 完成状态不只依赖颜色，必须同时显示图标和文字；
- 折叠计划、状态选项和完成按钮具备明确的键盘焦点与无障碍名称；
- 自动进入下一组后，将焦点移动到新组标题，并通过礼貌级别状态提示宣布变化；
- 动画遵循用户的减少动态效果设置；
- 保存失败和疼痛提示使用可被辅助技术读取的状态区域。

## 16. 验收与验证

### 16.1 新用户首页

- 全新浏览器状态访问 `/`，用户能在 5 秒内说明产品用途。
- 点击落地页主按钮进入 `/app/`。
- 用户打开 `/app/` 后能在 3 秒内指出从哪里开始训练。
- `/app/` 首屏只有一个视觉最强的主按钮。
- 不滚动即可看到今日方案和主按钮。
- 首屏不出现账号、AI、离线、安装四个常驻状态标签。
- 零数据状态不显示周目标、支持约定、完整日常表单或空洞察卡片。

### 16.2 快速开始与状态调整

- 当天无状态时，点击“开始训练”只经过异常疼痛确认，第二次点击进入第一组。
- 当天已有疼痛状态时，不重复询问。
- 未填写完整四项状态仍可使用明确标注的默认方案。
- 四项状态在一个手机屏幕内完成，并能改变训练类型、时长或强度说明。
- 明显疼痛产生恢复方案而不是普通负重训练。

### 16.3 逐组完成

- 模板预填次数完全不修改，点击完成后可以保存。
- 平板支撑 30 秒不填写重量可以保存。
- 快走 10 分钟不填写重量可以保存。
- 自重动作不填写重量可以保存。
- 输入数据但未点击完成的组不会进入正式记录。
- 已完成、待完成和已跳过状态均可见。
- 完成后自动进入下一组，并可撤销或返回修改。

### 16.4 结束与草稿

- 训练开始前不展示总结字段。
- 结束时才询问时长、整体感受和可选备注。
- 没有已完成组时不能保存空训练。
- 有未完成组时明确提示剩余数量。
- 刷新后恢复当前组、实际结果、组状态和开始时间。
- 保存成功后草稿被清除，正式历史记录只包含已完成组。

### 16.5 路由、离线与回归

- 本地服务器和 GitHub Pages 均可访问 `/` 与 `/app/`。
- PWA 安装后从 `/app/` 启动。
- 离线访问根路径显示落地页，离线访问应用路径显示应用页。
- 现有历史记录、模板、编辑、删除、导入导出和洞察数据不丢失。
- 桌面和常见手机尺寸均完成真实浏览器验证。

## 17. 最终 P0 用户旅程

> 认识产品 → 看到默认或状态调整后的今日方案 → 必要时确认异常疼痛 → 进入第一组 → 明确完成每一组 → 结束时补充整体感受 → 保存成功

实现时所有取舍都应服务这条旅程。任何不会让这条路径更清楚、更快或更可靠的新增内容，都不进入本轮 P0。
