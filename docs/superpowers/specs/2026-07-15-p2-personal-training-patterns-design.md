# P2 个人训练规律设计

日期：2026-07-15  
状态：用户已确认设计方向，待审阅规格

## 目标

在 P1 的“下一次训练”闭环稳定后，让用户在“进步”页看到少量、可解释、可行动的个人规律：哪些恢复信号与自己的训练完成和主观强度同时出现。P2 只陈述记录中的共同出现和趋势，不将其表达为因果、预测或医疗判断。

用户应能在半分钟内回答：

1. 我的记录已经足够得出什么初步观察？
2. 这条观察基于多少次记录，可信度如何？
3. 下次训练或恢复我可以做什么？

## 范围与非目标

### 包含

- “进步”页复盘中心下的一个“个人训练规律”卡片。
- 以同日训练与恢复记录配对，生成至多三条初步观察。
- 睡眠、疼痛/酸痛、精力与训练完成率、训练 RPE 的关系。
- 数据覆盖、样本量、缺失项和本地生成说明。
- 一条与 P1 下一次训练入口相衔接的行动建议。

### 不包含

- 相关系数、趋势折线图、预测、健康评分或自动周期化。
- 把“睡眠导致表现变化”等因果结论呈现给用户。
- AI 长报告、云端上传、付费墙、订阅权益或新账号数据。
- 动作级别的医学风险判断；高疼痛继续交给既有安全规则处理。

## 数据定义

### 有效观察日

有效观察日必须同时存在：

- 一条当天的训练记录；以及
- 至少一项恢复字段：睡眠、精力、酸痛或疼痛。

同一天多次训练合并为一个训练日：

- `completedSets`：当天明确完成的所有组数；
- `plannedSets`：当天 `completionSummary.total` 的总和；若历史记录没有该字段，则不把它计入完成率样本；
- `completionRate`：仅在存在计划组数时计算；
- `sessionRpe`：当天训练的平均整体 RPE；
- `workoutCount`：当天训练次数。

恢复值取当天保存的日常状态：`sleepHours`、`energy`、`soreness`、`pain` 和可选 `painArea`。备注、动作名称、重量、次数、精确时间和本地 ID 不进入规律结论或展示。

### 解锁门槛

- 低于 7 个有效观察日：不生成规律，只显示 `X/7 天`、缺少的恢复字段和训练记录提示。
- 7–9 个有效观察日：显示“初步观察”，每条都标注“样本较少”。
- 10 个及以上有效观察日：显示“已有一定依据”，仍不使用因果措辞。

每条规律都必须有至少两个样本的两组对比；不足时不显示该规律。最多展示三条，避免把数据噪声堆成报告。

## 可解释规则

将有效观察日按明确阈值切成两组，只有差异达到下列最小幅度时才展示。

| 观察 | 分组 | 最小差异 | 用户文案方向 |
| --- | --- | --- | --- |
| 睡眠与完成率 | 睡眠 < 6.5h / >= 6.5h | 完成率差 >= 15 个百分点，或平均完成组数差 >= 1 | 睡眠较足的记录中，训练完成更稳定/较少 |
| 睡眠与主观强度 | 睡眠 < 6.5h / >= 6.5h | 平均 RPE 差 >= 1 | 睡眠较少时，训练感觉更吃力/没有明显区别 |
| 疼痛与完成率 | 疼痛 >= 2 / < 2 | 完成率差 >= 15 个百分点，或完成组数差 >= 1 | 出现不适时，完成情况更容易受影响 |
| 酸痛与训练强度 | 酸痛 >= 4 / < 4 | 平均 RPE 差 >= 1 | 酸痛明显的日子，训练感觉更吃力 |
| 精力与完成率 | 精力 <= 2 / >= 4 | 完成率差 >= 15 个百分点，或完成组数差 >= 1 | 精力较高的日子，完成更稳定 |

正反方向都可显示。若差异不达标，则明确显示“目前没有足够一致的变化”，而不是强行给规律。

### 结论结构

每条卡片固定包含：

- 标题：例如“睡眠较足时，你的完成更稳定”。
- 证据：例如“基于 8 个同时记录训练和恢复的日子；睡眠 >= 6.5h 的 4 天平均完成 92%，其余 4 天为 70%。”
- 边界：例如“这是记录中的共同变化，不代表睡眠是唯一原因。”
- 行动：例如“下次睡眠不足时，优先使用轻量方案或先完成第一轮动作。”

高疼痛（>= 4）永远优先显示安全行动，而不是把它作为“训练表现规律”进行比较。

## 信息架构与交互

“个人训练规律”放在：

`进步 → 复盘中心 → 个人训练规律 → 28 天个人进展`

它比阶段报告更短、比每周复盘更个体化：

- 周复盘回答“本周发生了什么”；
- 个人规律回答“在你自己的记录中，哪些信号经常同时出现”；
- 28 天报告回答“最近阶段是否更稳定”。

卡片默认只展开首条最有行动价值的观察，其余两条使用“查看其余观察”展开；数据不足则显示进度、缺少项和“继续记录”的单一说明。规律卡不设置导出和分享按钮，避免低样本结论脱离上下文传播。

如果已有 P1 `nextWorkoutPlan`，行动按钮为“查看下一次训练”，回到“今天”页；否则为“开始今天训练”。不自动改写计划。

## 纯派生接口

不新增持久化字段。实现保持为纯派生函数：

- `buildTrainingPatternDataset()`：生成按天配对、已脱敏的计算数据；
- `buildPersonalTrainingPatterns()`：返回覆盖状态、观察与行动；
- `renderPersonalTrainingPatterns()`：只渲染派生结果；
- `patternObservation(...)`：封装分组、差异、样本门槛和安全文本。

既有 `dailyLogs`、`workouts`、`completionSummary`、`nextWorkoutPlan` 仅被读取。历史训练没有 `completionSummary` 时可用于 RPE 和训练次数，但不用于完成率比较。

## 安全、隐私与错误处理

- 固定显示“基于你的本机记录；不构成医疗诊断或因果结论”。
- 不展示 `painArea` 的具体文字；它只用于将不适记录计为已采集，不形成医疗推断。
- 没有合格样本、数值缺失或样本分布不足时跳过对应规则。
- 出现高疼痛时，优先显示既有恢复与专业咨询提示。
- 计算异常时回退到覆盖进度，不影响“今天”“训练”与既有周报。

## 验收标准

- 0–6 个有效观察日只显示进度与缺失项，不输出规律。
- 7 天后能按规则显示睡眠、疼痛/酸痛或精力的至少一条可解释观察。
- 每条观察都包含样本数、两组对比、非因果边界和行动建议。
- 高疼痛优先出现安全建议；数据不足或差异不足不制造结论。
- 用户可从行动入口回到 P1 下一次训练或今天训练。
- 不读取或展示备注、动作细节、重量、次数、具体日期或 ID。
- 自动测试覆盖正向、反向、样本不足、高疼痛和移动端无横向溢出；原有 P0/P1 测试持续通过。

## 自检

本规格把 P2 限制为“已记录数据的初步观察”，与周复盘、28 天报告和长期报告分工明确。它没有把统计相关性包装为因果、预测或医疗判断，也不新增需要同步或付费维护的数据系统。
