# 服务端权益与 AI 配额实施计划

## 目标

在不接入支付、不上传本地业务数据的前提下，为云端 AI 建议增加服务端验证的 Free/Pro 权益与原子月度配额。未配置权益服务的部署继续使用现有共享云端模式，静态部署继续保持完整本地能力。

## 实施顺序

1. 新增 Supabase 迁移：建立权益表、AI 配额事件表、严格 RLS、索引和仅限 `service_role` 的查询/预留/完成/释放 RPC。
2. 新增独立权益适配模块：校验环境配置，调用 Supabase RPC，严格规范化配额摘要，并为本地测试限制 OpenAI 基础地址。
3. 重构账号会话解析：把访问令牌验证、刷新令牌轮换、Cookie 更新与路由响应分离，供会话、权益和 AI 接口复用。
4. 新增 `GET /api/account/entitlements`，并把 `POST /api/advice` 改为原子执行“验证账号、预留额度、调用上游、完成或释放”。
5. 扩展健康接口，明确返回 `deployment_shared` 或 `account_quota`，但不暴露任何密钥。
6. 扩展账号面板：仅在服务端确认权益功能后显示层级、用量、剩余量和 UTC 重置日期；服务异常不猜测 Free，支付未开放时不出现购买按钮。
7. 配额耗尽、账号缺失或权益故障时保留本地建议回退，并刷新权益摘要；权益状态只保存在运行时变量中，不进入业务 `state` 或 `localStorage`。
8. 更新环境示例、部署清单、隐私说明、版本号与 PWA 缓存版本。
9. 扩展冒烟测试：模拟 Auth、Supabase RPC 和 OpenAI，覆盖 Free 三次成功、第四次上游前阻断、失败释放、并发不超额、刷新 Cookie、静态模式和桌面/390px UI。

## 验收条件

- 浏览器不能通过本地状态获得 Pro 或增加配额。
- 同一账号的并发预留不能超过月度限额。
- 上游失败和超时释放预留；成功响应包含完成后的配额摘要。
- 权益服务启用但不可用时返回 503，不静默降级或猜测层级。
- 未配置权益服务时现有共享 AI 行为不变；GitHub Pages 的本地模式不变。
- 登录、退出和配额变化不改写本地业务存储。
- `npm.cmd run check`、`npm.cmd run test:smoke` 和 `git diff --check` 全部通过。
