# 多位支持伙伴设计

## 目标

把单一“支持约定”升级为可并存的家人、朋友和教练支持关系。每位伙伴都有独立的关怀频率、支持方式、沟通边界、下一次关怀和本地回访，不需要保存姓名、联系方式、账号或聊天内容。

这为家庭版和教练版建立真实的领域模型：同一用户可以分别和家人维持陪伴关系、和教练维持计划执行关系，而不会把两种关系的隐私、提醒和反馈混在一起。

## 数据模型与迁移

在 `settings` 新增 `supportPartners` 数组，每项包含：

- `id`：仅本地的随机标识。
- `role`：`family`、`friend` 或 `coach`。
- `cadence`：`twice_weekly`、`weekly` 或 `biweekly`。
- `style`：`check_in`、`activity` 或 `accountability`。
- `boundary`：`no_pressure`、`no_advice` 或 `ask_first`。
- `nextDate`：ISO 本地日期。
- `checkins`：仅日期与 1–5 评分，最多 12 条。

旧的单伙伴字段在首次读取时转换为一个伙伴；没有旧约定时迁移为空数组。导入时白名单所有枚举、校验日期、去重 ID 与同日回访，并限制伙伴数量为 6，避免恶意备份扩大本地数据。旧字段继续在导出中保留一个版本周期，以便旧版本可读取最早的伙伴，但新界面只读取数组。

## 交互

首页支持约定区域在没有伙伴时引导“建立支持约定”；有伙伴时显示紧凑列表，每张卡展示对象类型、频率、下一次关怀和支持感。用户可新增、编辑、标记一次关怀或分享该伙伴的邀请。

同一对话框用于新增和编辑。完成关怀时，评分对话框只作用于刚刚点击的伙伴。计划日只会写入选择了“提醒我完成计划”的那位伙伴邀请；教练协作简报仍是独立的主动分享功能。

## 隐私与安全

伙伴之间不共享回访、关怀日期或分享内容。任何邀请仍排除健康、训练明细、备注、具体日期和回访分数；只有责任提醒型伙伴邀请可能包含星期级训练节奏。伙伴数组、回访和计划历史都只存于本机与 JSON 备份。

## 验收

- 旧单伙伴状态无损迁移为一个伙伴；空状态不创建虚假伙伴。
- 可建立、编辑并分别完成至少两位不同角色伙伴的关怀。
- 每位伙伴的日期和回访独立推进；分享邀请只反映被选伙伴。
- 恶意导入的 ID、枚举、日期与超量伙伴被过滤；JSON 备份保留全部合法伙伴。
- 桌面和 390px 移动端无横向溢出；完整冒烟测试覆盖迁移、独立性和隐私文本。
