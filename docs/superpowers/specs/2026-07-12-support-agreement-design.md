# 支持约定设计

## 产品目标

把一次性的关怀摘要升级为可持续的支持关系。用户明确告诉家人、朋友或教练“我希望怎样被支持、多久联系一次、什么方式不要做”，降低他人帮助时的猜测成本，也让应用从个人记录工具走向陪伴与协作产品。

## 方案取舍

首版采用本机保存的单个“支持约定”，而不是公开社区、聊天或账号邀请。公开社区需要内容审核，实时协作需要账号、权限和云端敏感数据托管；支持约定能在现有本地优先架构中验证协作需求，并为未来家庭版和教练版提供可迁移的领域模型。

## 用户体验

首页增加“支持约定”面板。未设置时说明价值并提供创建按钮；设置后显示支持对象、关怀频率、下一次关怀日期、希望得到的帮助和明确边界。用户可以编辑约定、标记“已关怀”并顺延下一日期，或生成不含健康记录的邀请文本。

设置对话框包含：支持对象类型、频率、希望获得的支持方式、需要避免的方式。邀请分享优先调用系统分享，失败时复制；邀请仅描述约定，不自动附加关怀摘要。关怀摘要仍由用户单独选择披露范围。

## 数据模型

支持约定存入现有 `settings`：

- `supportEnabled`: 是否已创建
- `supportRole`: `family`、`friend` 或 `coach`
- `supportCadence`: `twice_weekly`、`weekly` 或 `biweekly`
- `supportStyle`: `check_in`、`activity` 或 `accountability`
- `supportBoundary`: `no_pressure`、`no_advice` 或 `ask_first`
- `supportNextDate`: 下一关怀日期，ISO 本地日期

字段经过白名单与日期校验，因此旧备份可无损升级，异常导入不会注入自由文本。首版故意不保存对方姓名或联系方式。

## 日期规则

创建或更新约定后，从当天起按频率计算下一日期：每周两次为 3 天、每周一次为 7 天、每两周一次为 14 天。点击“已关怀”以当前日期和原下一日期中较晚者为基准顺延，避免过期约定被连续点击后仍停留在过去。

## 隐私、安全与错误处理

全部数据保存在本机并随 JSON 备份导出。邀请文本不包含状态、疼痛、训练、备注或记录标识。系统分享被取消不提示错误；其他失败回退复制。日期异常时自动重建，不阻断页面。

## 验收标准

- 可创建、编辑、保存和关闭支持约定。
- 首页准确显示下一关怀日期及到期状态。
- “已关怀”按频率顺延并持久化。
- 三种支持方式和边界生成对应邀请文案。
- 邀请文本不包含任何健康与训练记录。
- JSON 备份和导入保留约定；旧数据使用安全默认值。
- 桌面和 390px 移动端无横向溢出，现有完整冒烟测试继续通过。
