# 公开交付完整性设计

## 产品缺陷与路线判断

当前产品已经具备记录、训练执行、每周复盘、28 天进展、周目标校准、本地备份和隐私授权等个人价值闭环，但距离商业化仍有三类缺口。

### 公开交付与信任

GitHub Pages 已能发布静态应用，但仓库使用项目子路径 `/Daily-Workout-Record/`，PWA 清单和 Service Worker 仍假定应用位于域名根目录。这会造成：

- 安装后的启动地址和作用域指向 `https://gish-fucker.github.io/`，而不是实际应用。
- 清单图标请求域名根目录并返回 404。
- Service Worker 安装时预缓存错误根路径，导致离线能力无法可靠启用。
- 在线访问看似正常，但“安装、离线、更新”这些商业产品信任信号实际上不成立。

### 长期付费价值

未来 Pro 最有价值的方向仍是 90 天及年度比较、阶段目标、可控云端深度复盘和多设备恢复。它们解决的是长期连续性，而不是把基础记录和安全能力锁起来。

在服务端权益体系建立前，不展示假订阅、假支付或无法兑现的 Pro 状态。可以先完成底层设计和数据边界，但不提前承诺已经可购买。

### 账户与交易基础

正式收费仍需要经营主体、价格、退款与发票政策、商户账户、支付回调和支持联系方式。Apple Pay 只能作为支付入口之一，不能替代订单、签名回调和服务端权益。

## 方案比较

### 方案 A：先修复公开交付完整性

修复成本低，立即影响所有公开用户，也让后续 Pro 有稳定发布载体。它不会引入新的健康数据处理或不可兑现承诺。

### 方案 B：先做 Pro 功能预览

能表达未来价值，但当前没有账户、权益和真实价格，容易形成“看得见却买不到”的不完整体验，也无法验证付费转化。

### 方案 C：先建设账户和支付

是商业化必经之路，但缺少经营与商户资料时无法闭环。现在强行实现只能得到不可信的本地会员标记或假支付按钮。

采用方案 A。完成后再按“账户边界 -> 服务端权益 -> 90 天报告 -> 支付渠道”的顺序推进。

## 技术设计

### PWA 清单

`manifest.webmanifest` 的 `id`、`start_url`、`scope` 和图标全部改为相对清单地址解析：

- 根路径部署时解析为 `/`。
- GitHub Pages 项目部署时解析为 `/Daily-Workout-Record/`。
- 不写死仓库名或生产域名，保留 Render、自托管和本地开发兼容性。

### Service Worker

Service Worker 从 `self.registration.scope` 得到实际部署基址，再把相对应用外壳条目解析为完整 URL。

离线导航采用以下规则：

- 在线导航成功时，按实际请求 URL 缓存响应，不把隐私政策或条款错误覆盖到首页缓存键。
- 离线时优先返回该请求自身的缓存版本。
- 未命中时才回退到作用域内的 `index.html`。
- `/api/` 请求继续绕过缓存；静态 GitHub Pages 没有云端 AI 时，应用沿用本地建议。

### 发布流程

保留现有 GitHub Pages Actions 工作流，只上传 `public/`。代码提交并推送 `main` 后，由工作流发布。

公开地址为：

`https://gish-fucker.github.io/Daily-Workout-Record/`

## 安全与隐私

- 不新增分析脚本、Cookie、账号或远程数据收集。
- 清单和缓存修复只改变静态资源定位，不扩大 Service Worker 作用域。
- 作用域由实际 `sw.js` 位置限制，不申请域名根目录控制权。
- GitHub Pages 版本默认使用本地建议；云端建议不可用时不得影响记录、复盘和导出。

## 验收标准

- 本地根路径部署的首页、图标、清单、安装与离线重载继续工作。
- 清单所有 URL 在根路径和项目子路径下都解析到应用自身。
- Service Worker 预缓存不包含域名根目录硬编码。
- 访问隐私政策或使用条款不会覆盖首页的离线缓存。
- GitHub Pages 工作流成功，公开首页与版本化 JS 返回 200。
- 公开清单图标、启动地址和作用域均位于 `/Daily-Workout-Record/`。
- 线上首次加载无资源 404，刷新和核心本地记录流程可用。
- 现有完整浏览器冒烟测试和桌面、390px 手机布局继续通过。
